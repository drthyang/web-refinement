# Data Model

The canonical definitions are the TypeScript types under `src/core/**/types.ts`.
This document explains the conventions, the reasoning behind the shapes, and how
the pieces fit together. When code and doc disagree, the code is authoritative.

## Conventions

These hold everywhere in `src/core` unless a type says otherwise:

| Quantity | Unit / convention |
| --- | --- |
| Cell lengths | ångström (Å) |
| Cell angles | degrees at the data boundary; radians internally |
| Fractional coordinates | dimensionless; wrapped by symmetry expansion |
| Reciprocal lattice | crystallographic (no-2π): `a*·a = 1` |
| Phase factor | `exp[2πi(hx + ky + lz)]` (2π explicit) |
| Neutron scattering length `b` | femtometre (fm) |
| X-ray form factor `f` | electrons (dimensionless) |
| Magnetic moment | Bohr magnetons (μ_B) |
| `|Q|` | Å⁻¹, defined as `2π/d` |
| Timestamps in files | ISO-8601 strings (JSON-portable) |

All data types are **plain objects with no methods**. Operations live in sibling
pure-function modules. This guarantees lossless `JSON.stringify` round-trips and
clean transfer across the Web Worker boundary.

## Math primitives (`core/math/types.ts`)

- `Vec3 = readonly [number, number, number]` — interpretation (fractional vs
  Cartesian) is contextual.
- `Mat3 = readonly [Vec3, Vec3, Vec3]` — row-major.
- `Complex = { re, im }` — structure factors are complex.

## Crystal model (`core/crystal/types.ts`)

### `UnitCell`
The six parameters `a, b, c, α, β, γ`. Derived tensors (metric, reciprocal,
fractional↔Cartesian matrices) are **computed on demand**, never stored, so they
cannot drift out of sync with the parameters.

### `AtomSite`
Label, element (+ optional isotope, oxidation state), fractional `position`,
`occupancy ∈ [0,1]`, and `adp` (displacement parameters). ADP is a tagged union:

```ts
{ kind: "isotropic";  bIso: number }
{ kind: "anisotropic"; uAniso: [U11,U22,U33,U12,U13,U23] }
```

Both are fully supported: the structure factor applies the matching
Debye–Waller factor, refinement frees the **symmetry-allowed U-tensor modes**
for anisotropic sites, and [`crystal/adp.ts`](../src/core/crystal/adp.ts)
converts between them. A refinement typically starts isotropic and the user
switches to anisotropic ("ADPs: anisotropic" toggle) once the fit is stable,
seeding each site's spherical U from its current B_iso. Note `B_iso = 8π² U_iso`.

### `SymmetryOperation` and `SpaceGroup`
An operation is an affine map on fractional coordinates,
`x' = rotation·x + translation (mod 1)`, plus the Jones–Faithful `xyz` string for
CIF round-tripping. A `SpaceGroup` is, minimally, a **list of operations** — which
is exactly what a CIF `_symmetry_equiv_pos_as_xyz` loop provides — with optional
IT number and Hermann–Mauguin symbol as metadata. This keeps Phase 1 free of a
built-in 230-group table; that table becomes an optional lookup later.

### `StructureModel`
`id + name + cell + spaceGroup + sites`. The asymmetric unit is stored; the full
cell is generated by symmetry expansion at calculation time. **No magnetic
fields** — magnetism is layered separately.

## Diffraction data (`core/diffraction/types.ts`)

Two observation types, one engine (Phase 2 goal).

### Single crystal
```ts
SingleCrystalReflection = { h, k, l, iObs, sigma? }
SingleCrystalDataset    = { id, name, radiation, reflections[] }
```

### Powder
```ts
PowderPoint   = { x, yObs, sigma? }
PowderPattern = { id, name, xUnit, radiation, points[], wavelength? }
xUnit ∈ "twoTheta" | "dSpacing" | "q" | "tof"
```

`Radiation` is a tagged union (`neutron` | `xray` | `neutron-tof`) so the
calculator picks the right scattering table and knows whether a single
wavelength exists (TOF does not).

### Calculated side
`CalculatedReflection` keeps **nuclear and magnetic contributions separate**
(`fNuclearSq`, `fMagneticSq`) alongside `dSpacing`, `|Q|`, `multiplicity`, and
the applied `lorentzPolarization`. Separation is a hard requirement from the
brief: nuclear and magnetic components must always be independently reportable.

### Shared observation contract
Both datasets expose observations with optional `sigma`. The engine derives
weights `w = 1/σ²`, falling back to unit weights (single crystal) or `σ = √y`
(raw powder counts) when absent. This shared shape is what lets one
least-squares driver serve both — only the *calculated-data generator* and
*residual function* differ.

## Magnetic model (`core/magnetic/types.ts`)

Layered over a `StructureModel` by id; **imports no crystal-refinement code**.

- `MagneticMoment` — `siteLabel`, a `frame` (`crystallographic` | `cartesian`),
  and `components: Vec3` in μ_B. Storing the vector (not magnitude+angles) avoids
  a redundant, drift-prone representation; magnitude and direction are derived.
- `PropagationVector = Vec3` in reciprocal-lattice units; `(0,0,0)` = commensurate.
- `MagneticModel` — `structureId`, `propagation[]` (single k in Phase 5),
  `moments[]`, and optional `domainPopulations` (future multi-domain).

The perpendicular-moment projection `M⊥ = M − q̂ (M·q̂)` and the magnetic
structure factor are computed in functions, not stored on the model.

## Refinement model (`core/refinement/types.ts`)

- `RefinementParameter` — `id, label, kind, value, initialValue, min?, max?,
  fixed, group?, expression?, esd?`. `kind` (a `ParameterKind` tag) says what the
  parameter physically drives; `esd` is filled from the covariance matrix after a
  fit.
- `ParameterBinding` — connects a parameter id to a location in the domain model
  (`targetId` + optional `targetKey`), so the engine can stay crystallography-blind.
- `RefinementResult` — status, final values + esds by id, `AgreementFactors`, and
  a per-iteration `history`. See [REFINEMENT_ENGINE.md](./REFINEMENT_ENGINE.md).

## Project file (`core/project/types.ts`)

`ProjectFile` aggregates everything: `schemaVersion`, `metadata`, `structures[]`,
`magneticModels[]`, `datasets[]`, `parameters[]`, `bindings[]`, `lastResult?`.
Being plain JSON, it round-trips through `JSON.stringify`; that invariant is
tested in `core/project/project.test.ts`. Format details and migration policy are
in [PROJECT_FORMAT.md](./PROJECT_FORMAT.md).
