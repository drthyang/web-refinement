# Magnetic symmetry analysis — the two routes

To determine and refine a commensurate magnetic structure you need the allowed
**magnetic order parameters** for a given propagation vector **k**. There are two
standard, complementary routes, and the workbench is building **both** (the user
explicitly wants both — Jana2020 leans on irreps, FullProf on the magnetic space
group / basis vectors):

| Route | Also called | Tools that use it | Status here |
|---|---|---|---|
| **A. Magnetic space group** | Shubnikov / coordinate / basis-vector | FullProf, ISODISTORT, Bilbao k-SUBGROUPSMAG | **working — full subgroup lattice** (see below) |
| **B. Representation analysis** | irreps / small representations | Jana2020, BasIreps, SARAh | **working: any G_k at k = 0 (induced point-group irreps) + abelian G_k at any k, incl. irrep combinations → named isotropy subgroup**; projective small reps (non-abelian, k ≠ 0) next |

Both start from the **little group of k**, `G_k = { g ∈ G : R_gᵀ·k ≡ k (mod reciprocal lattice) }`
([`littleGroup`](../src/core/magnetic/magneticGroups.ts)).

## Route A — magnetic space group (coordinate / Shubnikov) ✅

The candidate magnetic space groups are the subgroups **M** of the grey
(paramagnetic) group `G_k·1'` that do not contain pure time reversal. Keeping
the translation lattice intact (type-IV / anti-translation groups are the
star-of-k follow-up), each M is a pair **(H, θ)**: a translationengleiche
subgroup `H ≤ G_k` **and** a time-reversal homomorphism θ: H → {±1}, with
index `[G_k·1' : M] = 2·|Ḡ_k| / |H̄|`.

**The full lattice** ([`magneticSubgroupLattice`](../src/core/magnetic/subgroupLattice.ts))
enumerates *every* such pair, k-SUBGROUPSMAG-style (Perez-Mato et al. 2015),
not only the maximal index-2 ones: all subgroups of the little co-group are
generated by breadth-first closure over generator sets (exhaustive for a
finite group), each gets the GF(2) θ-enumeration
([`generateMagneticCandidates`](../src/core/magnetic/magneticGroups.ts) — the
parent (type I) group plus one type-III group per index-2 kernel), and the
resulting magnetic groups are sorted by index and grouped into **conjugacy
classes under G_k** — conjugate candidates are the same ordering in
symmetry-equivalent **domains**, so the UI lists one representative per class
with its domain count. Verified counts: 2/m → 5 subgroups / 11 candidates,
mmm → 16 / 51; conjugate mirror subgroups of 4mm merge into one class with 2
domains ([`subgroupLattice.test.ts`](../src/core/magnetic/subgroupLattice.test.ts)).

**Presentation.** The UI groups the candidates into collapsible index sections
(only the maximal index-2 section starts open) because the correct workflow is
top-down: the symmetry of the ordered phase is *usually a maximal subgroup* of
the paramagnetic group (the Landau/k-SUBGROUPSMAG practice, ITA A1) — one
descends the lattice only when no maximal candidate fits the data. Within a
section, candidates whose sites can actually carry a moment sort first (the
moment-forbidden ones stay listed, greyed, for completeness), then type I
before type III, then by BNS number. When a candidate's identification needed
a basis transformation, the 3D view can draw its **standard-setting cell**
(e.g. the orthohexagonal C-centered cell of Cmcm-family subgroups) over the
parent cell, from the transformation (P, p) reported by the setting search.

For each candidate, the **symmetry-allowed moment directions** on a site are the
null space of `{ θ_g·det(R_g)·R_g − I }` over the site stabilizer
([`allowedMomentDirections`](../src/core/magnetic/allowedMoments.ts)) — these are
the refinable moment modes (the "coordinates"/basis vectors FullProf refines);
the UI badges each candidate with the total moment dimension on the selected
sites (a group that forbids any moment is greyed out).
This route drives the moment model + refinement
([`buildMagneticModel`](../src/core/magnetic/momentModel.ts)).

**Orbit splitting.** When `G_k` is smaller than the parent group, it can split a
site's crystallographic orbit; each split orbit is an **independent magnetic
sublattice** (no magnetic operation relates its moment to the others'). The
moment model decomposes every magnetic site's orbit into `G_k`-orbits and gives
each its own allowed-moment basis, amplitudes ("`Mn1 orbit 2 M …`"), and moment
entry anchored at the orbit's representative position — the structure factor and
the 3D viewer expand each entry over the magnetic operations from that anchor.
A split orbit whose allowed dimension is 0 genuinely carries no moment (the
viewer draws no arrow there and says why). Tested in
[`orbitSplitting.test.ts`](../src/core/magnetic/orbitSplitting.test.ts).

**Moment size across sublattices.** Every allowed mode is normalized to unit
**Cartesian** length before parameters are built, so one unit of amplitude is
1 µ_B along the mode whatever the cell metric (a raw null-space vector like
(−1,1,0) is √3 µ_B long in a hexagonal cell — Mn₃Ga/Cm'cm' previously seeded
one sublattice √3× larger than the other). The refinable quantities are the
per-mode amplitudes themselves: only what the magnetic symmetry allows is
refined, and nothing beyond it is imposed. (An earlier equal-|m| constraint —
one shared magnitude per site with per-orbit direction angles — was removed:
same-size moments across sublattices are a physical prior, not a symmetry
requirement, and the fit should be free to disagree.)

**Report export.** Step 5's "Export report" button downloads a self-contained
HTML report of the current magnetic model: a projected structure figure
(down c*, moment arrows per sublattice, ⊙/⊗ for out-of-plane components,
magnetic supercell for k ≠ 0) plus the moment-parameter, cell, and
sublattice tables — built by
[`magneticReportHtml`](../src/core/export/magneticReport.ts) from the same
expansion that drives the 3D viewer.

**Tested:** k=0 candidate set (P2₁/m → P2₁'/m'), allowed-moment dimensions, the
simple-AFM recovery ([`magneticSimpleAfm.test.ts`](../src/core/workflow/magneticSimpleAfm.test.ts)).

Each candidate now also carries its standard **BNS/OG label** when it matches a
tabulated group in the standard setting (see below).

## Route B — representation analysis (irreps) 🚧 abelian route working

The magnetic representation `Γ_mag = Γ_perm(k) ⊗ Γ_axial` on the magnetic atoms
decomposes into the irreps of `G_k`; each irrep that appears carries a set of
**basis vectors** (magnetic modes), and refining a structure means refining the
amplitudes of one (or a few) irreps. This is the Jana2020 / BasIreps route.

**Implemented + tested** ([`magneticRepresentation.ts`](../src/core/magnetic/magneticRepresentation.ts)):
- `axialCharacter(R) = det(R)·tr(R)` — a moment is an *axial* vector (E→+3,
  2-fold→−1, mirror→−1, inversion→+3).
- `magneticRepresentationCharacter(structure, k, sites, G_k)` — the character
  `χ_mag(g) = χ_axial(R_g)·Σ_{atoms fixed mod lattice} e^{2πi k·L}` (Bertaut 1968).
- `irrepMultiplicity(reducible, irrep) = (1/|G|) Σ χ·χ*` — the standard
  decomposition. Verified against hand-computed cases: P1 → 3× trivial (3 DOF),
  C2 → A + 2B (Mz = A, Mx,My = B), inversion → 3 Ag (0 Au).

**Now implemented + tested** ([`irreps.ts`](../src/core/magnetic/irreps.ts)):
- **Abelian little-group irrep generator** — `abelianIrreps(G_k)` builds the 1-D
  irreps of any abelian little co-group *by construction* (no tabulated data):
  it verifies the group is abelian, decomposes it as a direct product of cyclic
  groups ⟨g₁⟩ × … × ⟨gₘ⟩ (greedy independent generators of maximal order), and
  enumerates χ_{p₁…pₘ}(g₁^{a₁}…gₘ^{aₘ}) = ∏ exp(2πi·pₖ·aₖ/nₖ). Covers
  triclinic/monoclinic/orthorhombic, the cyclic trigonal/tetragonal/hexagonal
  rotation groups, and any general-position little group, at any k including Γ.
- **Decomposition** — `decomposeMagneticRepresentation` gives which irreps carry
  the order and their multiplicities (Γ_mag = Σ nᵢ Γᵢ), and flags when the
  multiplicities come out non-integral (the signal that a non-symmorphic
  BZ-boundary k needs the projective small reps rather than the ordinary irreps).
- **Basis-vector projection** — `projectIrrepModes` applies the axial projection
  operator restricted to the reference site, Σ_{g fixes r₀} χ_Γ(g)*·e^{2πik·L}·
  det(R_g)R_g, giving the refinable moment directions each irrep allows. Verified:
  P1 → 3×trivial, C2 → A(Mz) + 2B(Mx,My), inversion → 3 Ag, D2 → three non-trivial
  irreps, C4 → the ±i pair; the UI shows the decomposition in the Magnetic panel.

**Now implemented + tested — non-abelian point-group irreps at Γ**
([`pointGroupIrreps.ts`](../src/core/magnetic/pointGroupIrreps.ts)): the irreps
of **any** crystallographic co-group (6/mmm, 4/mmm, m-3m, …), constructed —
not transcribed — by Clifford-theory induction along a chain of prime-index
normal subgroups (crystallographic point groups have order 2^a·3^b, hence are
solvable): fixed irreps extend in p ways via a Schur-average intertwiner
normalized by (cU)^p = ρ(g^p); orbits of size p induce a p·d-dimensional
irrep. The output is verified before use (Σ dim² = |G|, character
orthogonality — an exception rather than a wrong table). At **k = 0** these
ordinary irreps are exactly the small representations even for non-symmorphic
groups (every e^{2πik·τ} = 1 — Bradley & Cracknell Ch. 3), so the irrep route
now works at Γ for **every** parent — e.g. Mn₃Ga's 6/mmm gives
Γ_mag = 12 irreps (8×1-D + 4×2-D), and the eight real 1-D irreps map
one-to-one onto the eight maximal Shubnikov candidates (asserted in
[`pointGroupIrreps.test.ts`](../src/core/magnetic/pointGroupIrreps.test.ts)).
Verified against the classical tables: 4mm → [1,1,1,1,2], 432 → [1,1,2,3,3],
m-3 → [1×6, 3×2] with the complex Eg pair, m-3m → [1,1,1,1,2,2,3,3,3,3].
Multidimensional irreps are displayed with their modes; selecting them for a
*combination* is gated (kernel-vs-epikernel needs an order-parameter-direction
UI — the stabilizer bridge below already handles it representation-agnostically
once directions can be chosen). Reference: J.-P. Serre, *Linear
Representations of Finite Groups* (Springer, 1977), §8–9.

**Now implemented + tested — irrep combinations → isotropy subgroup**
([`isotropy.ts`](../src/core/magnetic/isotropy.ts)): the SARAh workflow (Wills
2000) of refining a **combination of irreps**, and the bridge back to Route A.
Selecting one irrep (the Landau single-irrep prescription) or several builds
the explicit moment configuration `m(j) = Σ_ν Σ_λ c_{νλ}·ψ_{νλ}(j)` with
**generic amplitudes** (reference-site modes transported over each site's
orbit by coset representatives, Izyumov-style), and its actual symmetry group
is computed as the **exact stabilizer** in `G_k × {1, 1'}` — the isotropy
subgroup of the order parameter (Stokes & Hatch 1988; the ISODISTORT
mode-amplitude ↔ symmetry correspondence, Campbell et al. 2006). Two
independent generic draws are intersected so no accidental invariance of one
draw survives. The stabilizer's operation set is then **identified against the
standard table** — this is how the app names the OG/BNS group implied by "the
ordering transforms as these irreps", with no symbol ever guessed. For 1-D
irreps every operation acts by the scalar χ(g), so the result is
amplitude-independent (kernels; epikernels need multidimensional order
parameters) — asserted in the tests. Verified: each real irrep of P2/m on a
general site reproduces exactly its θ = χ Shubnikov candidate; Ag ⊕ Bg → P-1
(BNS 2.4); Ag ⊕ Au → P2 (BNS 3.1); all four irreps → P1 (BNS 1.1); the
P2₁/m Bg-type irrep names **P2₁'/m'**, the Mn₃Ga ground state
([`isotropy.test.ts`](../src/core/magnetic/isotropy.test.ts)).

Honest failure modes (each reported with a reason in the UI): a selection that
carries no moment on the chosen sites; conjugate-pair (complex) irreps, whose
real combinations generally stabilize as type-IV groups; complex phase factors
`e^{2πik·L} ∉ {±1}`; a stabilizer needing a primed pure translation (type IV).

**Remaining:**
1. **Order-parameter directions for multidimensional irreps** (kernel vs
   epikernels): the modes are shown, but combination-selection is gated until
   a direction picker exists; the stabilizer construction above then names the
   groups unchanged.
2. **Projective small representations** for non-abelian little co-groups at
   BZ-boundary k ≠ 0 (phase factors `e^{-2πi k·τ}`) — Bradley & Cracknell
   Ch. 3–4. The integer-multiplicity check already detects when they are
   needed; at Γ the induced ordinary irreps are exact.
3. Conjugate-pair (complex-irrep) combinations and their type-IV isotropy
   groups — needs the type-IV table rows (see below) and Fourier-coefficient
   stabilizers.

## BNS / OG labels ✅ (bundled standard table)

Candidates now carry the standard **Belov–Neronova–Smirnova (BNS)** and
**Opechowski–Guccione (OG)** numbers and symbols (e.g. `P2₁'/m'`, BNS 11.54, OG 11.5.63)
whenever their operation set matches a tabulated Shubnikov group in its standard
BNS setting; otherwise `standard` is `null` and the descriptive primed-operation
label is kept (**a wrong symbol is worse than none** — nothing is guessed).

- **Data** ([`bnsOgTable.ts`](../src/core/magnetic/bnsOgTable.ts), generated by
  [`gen_bns_og_table.py`](../scripts/gen_bns_og_table.py)): the ISO-MAG magnetic
  group data (ISOTROPY Software Suite, stokes.byu.edu — compiled from Litvin,
  *Magnetic Group Tables*, IUCr 2013, and Bradley & Cracknell 1972), taken from
  pymatgen-core's `symm_data_magnetic.sqlite` redistribution. Types I (230) and
  III (674) are bundled — exactly what the k = 0 / little-group θ-enumeration
  can produce. **No label is hand-transcribed**; the generator verifies every
  entry (identity unprimed, correct primed count per type, group closure with
  homomorphic time reversal, unique operation-set signature) before emitting.
- **Lookup** ([`bnsOg.ts`](../src/core/magnetic/bnsOg.ts)):
  `identifyMagneticGroup(ops)` canonicalizes the operation set (rotation +
  translation mod lattice + time reversal) into a signature and matches it
  exactly. For types I–III the OG symbol equals the BNS symbol (asserted at
  generation time); the OG *number* differs and is reported alongside.
- **Tested** ([`bnsOg.test.ts`](../src/core/magnetic/bnsOg.test.ts)): full-table
  round trip (all 904 groups re-identified from their own operations), the four
  P2₁/c candidates against their BNS/OG numbers, the P2₁'/m' Mn₃Ga ground state,
  bidirectional completeness (θ-enumeration from parents 14/216/225 yields
  *exactly* the tabulated type-I/III groups of those parents), and graceful
  `null` for a non-standard setting.

**Setting-transformation search ✅**
([`identifyMagneticGroupAnySetting`](../src/core/magnetic/bnsOg.ts)): a proper
subgroup of a standard-setting parent is often expressed in a non-standard
setting (2-fold along a instead of b, off-origin inversion center, …), so the
exact lookup rightly misses. The search conjugates the operation set by the
ITA basis transformation `R′ = P⁻¹RP, t′ = P⁻¹(Rp + t − p)` (ITA Vol. A §1.5)
over the 24 proper signed axis permutations **and the three orthohexagonal
C-centered cells (det P = 2, composed with the permutations)** × ¼-grid origin
shifts, and matches each transformed set **exactly** — the identification
stays exact, only the setting is searched, and the transformation used is
reported alongside the label (e.g. `(a, a+2b, c; 0, 0, 0)`). det P = 2 uses
integer adjugate arithmetic (a basis applies only when adj·R·P is divisible
by det) and expands the centring cosets {P⁻¹v mod 1} before matching — this
is how the **orthorhombic and monoclinic subgroups of hexagonal parents**
(Cmcm, C2/m, … of P6₃/mmc) get their standard symbols. Wired into the
subgroup lattice (class representatives) and the isotropy result.
Rhombohedral↔hexagonal (det 3) and monoclinic cell choices 2/3 are outside
the family and still return `null`. Tested: a-unique P2/m recovered,
off-origin P-1 recovered with its ¼ shift, primed operations preserved,
honest `null` for a rhombohedral 3-fold, and Cmcm (#63) / C2/m (#12) /
type-III orthorhombics recovered from the P6₃/mmc lattice
([`bnsOgSetting.test.ts`](../src/core/magnetic/bnsOgSetting.test.ts)).

Not covered yet: type-II (grey) and type-IV (anti-translation) groups — the
generator does not produce them today (they become relevant with the star of k /
anti-translation work, M4); the generation script can bundle them by widening
`MAGTYPES`.
